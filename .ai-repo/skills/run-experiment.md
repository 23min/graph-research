# Skill: Run Experiment

Execute an EXP-NN through the bench harness with full reproducibility and produce ready-to-review results.

## Pre-run checks (refuse to run if any fails)

- [ ] `docs/experiments/EXP-NN-slug/hypothesis.md` exists and has a falsification condition.
- [ ] `docs/experiments/EXP-NN-slug/setup.md` has all four reproducibility fields filled: seed, config path, code SHAs, split version hash.
- [ ] The configuration file referenced in setup.md exists and is committed.
- [ ] The fixtures exist at their declared paths.
- [ ] The current dag-map + bench SHAs match (or can be updated to match) what setup.md expects.
- [ ] `docs/experiments/EXP-NN-slug/results/` is empty or only contains a `.gitkeep`.

If any check fails: print the failing check, do not run, and route back to `design-experiment` for remediation.

## Run

### Baseline pass

```
node bench/run-experiment.mjs --exp EXP-NN --config baseline
```

Writes to `results/baseline/`:
- `metrics.csv` — per-fixture vector fitness components.
- `metrics.json` — same, structured.
- `renders/<fixture-id>.svg` — per-fixture rendering.
- `renders/<fixture-id>.png` — rasterized for contact sheet.
- `provenance.json` — the four reproducibility fields at run time.

Every output file carries a header:
```
# generated by bench/run-experiment.mjs at <code-sha> seed=<seed> split=<split-hash>
```

### Intervention pass

```
node bench/run-experiment.mjs --exp EXP-NN --config intervention
```

Same outputs under `results/intervention/`.

### Additional baselines (if declared in setup.md)

One invocation per additional baseline, each writing to its own `results/<baseline-name>/` subfolder.

### Diff and aggregate

```
node bench/diff-experiment.mjs --exp EXP-NN --against baseline
```

Writes:
- `results/diff-table.csv` — per-fixture per-metric delta.
- `results/diff-table.md` — human-readable version.
- `results/diff-plates/<fixture-id>.svg` — side-by-side baseline vs. intervention.
- `results/contact-sheet.pdf` — 4×4 grid of all fixtures, baseline-left / intervention-right.

### Aggregate statistics

In `results/summary.md`, produce:

- Per-metric mean, median, IQR across fixtures.
- Effect size (Cohen's d for paired design) with 95% bootstrap CI.
- Per-fixture win/tie/loss count.
- Check of the pre-committed falsification condition: does the observed effect clear the bar?

## Outcome classification

Based on the falsification condition pre-committed in `hypothesis.md`:

- **Positive:** effect size exceeds the pre-committed threshold, CI excludes the null, at least 70% of fixtures improve.
- **Negative:** effect size below the null threshold, CI crosses zero, or majority of fixtures regress.
- **Superseded:** a newer EXP has supplanted this one; mark as such and cross-reference.
- **Inconclusive:** signal is in the grey zone — neither clearly positive nor clearly negative. Record the specific reason (underpowered, confound found, noisy fixtures).

Do **not** massage the falsification condition to match the result. If the result doesn't cleanly meet the pre-committed bar, classify as inconclusive and write up.

## Draft discussion.md

Populate the `discussion.md` placeholder with sections:

### 1. Outcome (with status banner)

```markdown
## Outcome

Status: <positive | negative | superseded | inconclusive>
```

### 2. Headline

One sentence with effect size and CI.

### 3. Detailed findings

Per-metric breakdown, per-fixture category breakdown, any unexpected patterns.

### 4. Caveats and scope

What the result does not say. Confounds. Fixture-class limits. Generalization boundaries.

### 5. Follow-up

If positive: what's the next experiment that builds on this?
If negative: what does this close off?
If inconclusive: what additional data or analysis would resolve it?

### 6. Track 3 recommendation

- **Post-worthy?** Yes / no / maybe.
- **Post type recommendation:** deep-dive / takeaway / negative-result candor / N/A.
- **Deferred?** If recommended but held, the reason (needs more fixture coverage, principal review, etc.).

## Update the experiments index

```markdown
| EXP-NN | <slug> | closed-positive | <one-line headline> | {YYYY-MM-DD} |
```

## Commit etiquette

This skill does not commit. The principal reviews the results and discussion, approves, and commits manually. The bench is strict about the research-discipline rule: no unauthorized commits.

## Invocation pattern

> Run EXP-NN.

Produces: results/, discussion.md drafted, index updated. Does not commit. Does not publish content. Does not close the EXP without principal review.

## What this skill does NOT do

- Design the experiment. That is `design-experiment`.
- Write a post. That is `draft-content`.
- Garden the docs. That is `doc-gardening-research`.
- Commit or push. The principal approves commits explicitly.
